Site Hierarchy & Roles
6 / 10 difficulty
What it is

Site Hierarchy defines how MECM is physically and logically organized — how many sites you have, how they relate to each other, and which server roles handle what. Every MECM action flows through this structure: clients check in with their Management Point, pull content from their Distribution Point, and receive updates from the Software Update Point. Design this correctly before anything else — bad hierarchy decisions are expensive to fix.

Lifecycle placement
Pre-stage

Design hierarchy, plan SQL instance, allocate VMs, extend AD schema

Stage

Install Site Server + SQL, configure MP and DP, verify site status

Deploy

Add SUP, Reporting Point, assign clients via boundary groups

Verify

Component status green, clients registering, MP publishing to AD

Core roles
Site Server
SS
The brain. Processes all site operations. One per site.
Management Point
MP
Clients talk here for policy and inventory. Required role.
Distribution Point
DP
Holds content. Clients pull from here. One per subnet ideally.
Software Update Point
SUP
Wraps WSUS. Manages patch sync and deployment.
SQL Server
SQL
Stores site DB. Separate VM in production.
Reporting Services Point
RSP
Hosts SSRS reports. Add after site is stable.
Install order
STEP 0
Patch all VMs fully before touching the installer. Standing rule on every build.
STEP 1
Extend AD schema — run extadsch.exe from MECM media. Requires Schema Admin. One-time.
STEP 2
Create System Management container in AD, delegate Full Control to the Site Server computer account.
STEP 3
Install SQL Server 2022 on SQL VM. Collation must be SQL_Latin1_General_CP1_CI_AS.
STEP 4
Install Windows ADK + WinPE add-on on the Site Server.
STEP 5
Run Prerequisite Checker. Resolve all errors before proceeding.
STEP 6
Install Primary Site via setup wizard. Set site code (3 chars — cannot be changed after).
STEP 7
Verify: Monitoring → Site Status → all green. Check sitecomp.log.
STEP 8
Add Distribution Point role. Configure boundary groups.
STEP 9
Install SUP role. Install WSUS first, do not configure it — let MECM manage it.
Key checks
✓Monitoring → Site Status → Component Status — all green
✓System Management container in AD with correct permissions
✓DP shows no errors — check distmgr.log
✓SUP first sync completes — check wsyncmgr.log
✓Test client registers — check ccmsetup.log on client
Troubleshooting
AD_SCHEMA
MP can't publish — verify System Management container and Full Control delegation. Check mpcontrol.log.
SQL_COLLATION
Install fails — must be SQL_Latin1_General_CP1_CI_AS. Wrong collation = rebuild SQL.
PREREQ_FAIL
Check ConfigMgrPrereq.log at C:\ root after failed install.
RED_COMPONENT
Right-click → Show Messages. Logs at C:\Program Files\Microsoft Configuration Manager\Logs\.
MP_COMMS
Clients not checking in — verify firewall allows 80/443/10123. Check mpcontrol.log.
Q&A — click to reveal
Q1 — Where in the console do you check site component health?▾
Monitoring workspace → Site Status → Component Status. Green = OK, yellow = warning, red = critical.
Q2 — Log file for Management Point communication?▾
mpcontrol.log on the site server. Also MP_Relay.log for client message relay issues.
Q3 — Required SQL collation and why?▾
SQL_Latin1_General_CP1_CI_AS — MECM's DB schema is built against it. Cannot be changed after install without rebuilding SQL.
Q4 — What AD container must exist for the MP to publish?▾
System Management container under System in AD. Site Server computer account needs Full Control delegated. Without it clients can't discover the MP via AD.
Q5 — What is a site code and can it be changed?▾
3-character alphanumeric identifier unique to the site (e.g., P01). Cannot be changed after installation.
Q6 — What must be installed before the SUP role?▾
WSUS must be installed first. Install it but do not configure it — let MECM control WSUS configuration.
Q7 — Default log file location on the site server?▾
C:\Program Files\Microsoft Configuration Manager\Logs\. Use CMTrace.exe. Client logs at C:\Windows\CCM\Logs\.
Q8 — Difference between Primary and Secondary Site?▾
Primary Site has its own SQL DB and manages clients directly. Secondary Site uses parent Primary's DB — used for remote locations with limited WAN.
Q9 — Tool to run before MECM installation?▾
Prerequisite Checker — run standalone as prereqchk.exe. Logs to ConfigMgrPrereq.log at C:\ root.
Q10 — Two Windows components required on Site Server before setup?▾
Windows ADK and the WinPE Add-on for Windows ADK. Must match your MECM version.

Client Installation Methods
5 / 10 difficulty
What it is

The MECM client (CCMSetup.exe) must be installed on every managed device. How it gets there depends on your environment. Client Push is the most common but requires firewall exceptions and admin rights on target. GPO is reliable for domain-joined machines at scale. Task Sequence is best for OSD scenarios. Understanding the right method prevents wasted troubleshooting time chasing silent failures.

Lifecycle placement
Stage

Select install method, configure prerequisites (firewall, WMI, admin share)

Deploy

Push or trigger client install, monitor in console

Verify

Client appears in Assets and Compliance, heartbeat discovery running

Methods
Client Push
Most common
MECM pushes CCMSetup.exe via admin$ share. Requires firewall rules and local admin account.
GPO / Logon Script
Scale method
Deploy CCMSetup.exe via GPO startup script. Good for large AD environments.
Manual / Script
Targeted
Run CCMSetup.exe directly or via PowerShell. Useful for servers or troubleshooting.
Task Sequence
OSD only
Client installed as part of OS deployment. Built into the OSD task sequence automatically.
Software Update
Upgrade only
Upgrade existing clients via WSUS/SUP. Not for initial install.
Client push — step by step
STEP 1
Admin → Site Configuration → Sites → Client Installation Settings → Enable Client Push
STEP 2
Configure the Client Push Installation account — domain account with local admin rights on targets
STEP 3
Ensure firewall allows: File/Printer Sharing (TCP 445), WMI (TCP 135 + dynamic), RPC
STEP 4
Discover devices first via AD System Discovery or Heartbeat Discovery
STEP 5
Right-click device in console → Install Client
STEP 6
Monitor: ccm.log on site server, ccmsetup.log on target client
Key checks
✓Client shows in Assets and Compliance → Devices with green checkmark (Active)
✓Heartbeat Discovery record exists — DDRs sending every 7 days by default
✓Client properties show correct site code assigned
✓Hardware inventory running — check Last Hardware Inventory date in console
Troubleshooting
PUSH_FAILS
Check admin$ share accessible, WMI running, firewall exceptions in place. Check ccm.log.
WON'T_REGISTER
Boundary group issue — client can't find its MP. Check boundaries and locationservices.log.
WRONG_SITE
Boundary group overlap. Check clientlocation.log for site assignment logic.
GOES_INACTIVE
Heartbeat DDR not arriving — check inventoryagent.log and network path to MP.
Q&A
Q1 — Log file to check on the target machine during client push?▾
ccmsetup.log at C:\Windows\ccmsetup\Logs\. First stop for any client install failure.
Q2 — Log file on the site server for client push operations?▾
ccm.log in the MECM Logs folder. Shows push initiation, failures, and retry attempts.
Q3 — What account does Client Push use and what rights does it need?▾
Client Push Installation account — domain account configured in Site Settings. Needs local administrator rights on every target device.
Q4 — Where do you enable Client Push in the console?▾
Administration → Site Configuration → Sites → right-click site → Client Installation Settings → Client Push Installation tab.
Q5 — What ports must be open for client push?▾
TCP 445 (SMB/admin$ share), TCP 135 (RPC endpoint mapper), plus dynamic high ports for WMI/RPC. Also TCP 80/443 for client-to-MP post-install.
Q6 — How does a client know which site to register with?▾
Via Boundary Groups — client's IP/subnet matched to a boundary, which maps to a boundary group, which points to a site and MP.
Q7 — What is a DDR and what sends it?▾
Data Discovery Record — sent by client via Heartbeat Discovery every 7 days by default. Keeps the client record active in the CM DB.
Q8 — Best method for installing the client on 500 new domain-joined PCs?▾
GPO startup script or Client Push with automatic discovery. GPO is more reliable at scale and handles machines offline during the initial push window.
Q9 — Where do you see if a client is active in the console?▾
Assets and Compliance → Devices. Client column shows green checkmark (Active) or blank. Activity column shows Last Active timestamp.
Q10 — What log shows why a client was assigned to the wrong site?▾
clientlocation.log on the client. Shows the full site assignment logic — which boundaries were evaluated and which MP was selected.

Collections
4 / 10 difficulty
What it is

Collections are the targeting mechanism for everything in MECM — software deployments, patches, task sequences, compliance baselines, and client settings all target a collection. A collection is a group of devices or users defined by rules. Bad collection design deploys software to machines that shouldn't have it, or misses machines that should. Simple concept, dangerous when poorly structured.

Lifecycle placement
Stage

Design collection hierarchy, create device/user collections, define membership rules

Deploy

Target all deployments — software, patches, task sequences — to collections

Verify

Confirm correct members, check update schedules, validate no overlap

Collection types & rules
Device Collection
Most used
Groups computers/servers. Target for software, patches, OSD.
User Collection
User-centric
Groups users. Target for user-available apps in Software Center.
Direct Rule
Static
Manually added devices. Never auto-updates.
Query Rule
Dynamic
WQL query auto-populates membership on schedule.
Include Rule
Nesting
Pulls in all members of another collection.
Exclude Rule
Nesting
Removes members of another collection. Good for exceptions.
Create a query-based collection
STEP 1
Assets and Compliance → Device Collections → right-click → Create Device Collection
STEP 2
Set Limiting Collection — always set to an appropriate parent. Never limit to All Systems for production.
STEP 3
Add Membership Rule → Query Rule → Edit Query Statement
STEP 4
Write WQL: SELECT * FROM SMS_R_System WHERE OperatingSystemNameandVersion LIKE '%Server%'
STEP 5
Set Update Schedule — default 7 days. For critical collections, set to 1 day or use incremental updates.
STEP 6
Preview collection members before saving — confirm query returns expected results.
Key checks
✓Member count matches expectation — right-click collection → Show Members
✓Limiting collection is not too broad — avoid targeting All Systems in production
✓Update schedule set appropriately for deployment urgency
✓No circular references between Include/Exclude rules
Troubleshooting
MISSING_MEMBERS
Query rule not returning devices — check WQL syntax, verify hardware inventory is current.
STALE_MEMBERS
Device removed from AD still in collection — update schedule hasn't run or incremental updates disabled.
WRONG_DEPLOYMENT
Device in wrong collection — check all query/include rules. Use Deployment tab on device.
PERFORMANCE
Too many incremental-update collections hammers SQL. Microsoft recommends max 200 per site.
Q&A
Q1 — What is a Limiting Collection and why does it matter?▾
The parent boundary restricting collection membership. A query collection can only contain devices already in its Limiting Collection. Misconfigured limiting collections cause deployments to miss or over-include devices.
Q2 — What query language does MECM use for collection rules?▾
WQL — WMI Query Language. Similar to SQL SELECT syntax. Example: SELECT * FROM SMS_R_System WHERE Name LIKE 'SRV%'
Q3 — Where do you preview collection membership before saving?▾
In the collection wizard after writing the query rule, click Preview. Shows expected members based on current inventory data.
Q4 — Difference between Direct Rule and Query Rule?▾
Direct = static, manually added, never auto-updates. Query = dynamic, auto-populates via WQL on schedule. Use Direct for exceptions, Query for everything else.
Q5 — What is Incremental Update and when should you use it?▾
Evaluates only recently changed records. Faster but resource-intensive at scale. Use for high-priority collections. Cap: ~200 incremental collections per site.
Q6 — How do you exclude specific devices from a collection?▾
Add an Exclude Collection rule pointing to a collection containing the devices to exclude. Members of excluded collection are removed even if they match the query rule.
Q7 — Where do you see which deployments are targeting a device?▾
Assets and Compliance → Devices → right-click device → open properties → Deployments tab.
Q8 — Built-in collection to target all managed clients?▾
All Systems (device) and All Users. Built-in, cannot be deleted. Never deploy production software directly to All Systems.
Q9 — How do you force a collection to update membership immediately?▾
Right-click the collection → Update Membership. Forces immediate evaluation outside the schedule.
Q10 — What console node holds Device Collections?▾
Assets and Compliance workspace → Device Collections.

Software Distribution
5 / 10 difficulty
What it is

Software Distribution covers getting applications, scripts, and packages onto managed devices. MECM has two models: Applications (modern, detection-method based, supports user-targeting and supersedence) and Packages/Programs (legacy, script-based). Applications are preferred. The deployment process has three phases: create the content object, distribute content to DPs, then deploy to a collection. Miss any phase and the deployment silently fails.

Lifecycle placement
Stage

Create Application/Package, set detection method, configure deployment types

Deploy

Distribute content to DP, create deployment to target collection

Verify

Monitor deployments, check enforcement state, review failure reasons

Application vs Package
Application
Modern — use this
Detection method determines install state. Supports requirements, dependencies, supersedence, user installs.
Package/Program
Legacy
Script-based. No detection method. Use only for simple scripts or legacy software.
Detection Method
Critical
Registry key, file/folder, or MSI product code. Tells MECM if app is installed. Get this wrong and apps re-install constantly.
Deployment Type
Install logic
Actual install command, detection method, requirements, and user experience settings within an Application.
Requirements
Targeting filter
OS version, disk space, architecture rules. MECM evaluates before installing.
Supersedence
Upgrades
Links new version to old. Auto-uninstalls old and installs new during deployment.
Deploy an application
STEP 1
Software Library → Application Management → Applications → Create Application
STEP 2
Set install command (e.g., setup.exe /S), uninstall command, and detection method
STEP 3
Configure User Experience: install for System or User, Software Center behavior, restart behavior
STEP 4
Distribute Content: right-click Application → Distribute Content → select DP or DP Group
STEP 5
Wait for content distribution — check Monitoring → Distribution Status
STEP 6
Deploy: right-click Application → Deploy → select collection, purpose (Required vs Available), schedule
STEP 7
Monitor: Monitoring → Deployments → select deployment → view status by device
Key checks
✓Content distribution completed — Monitoring → Content Status shows all DPs green
✓Detection method validated — right-click app → Simulate Deployment on test machine
✓Deployment shows in target device's Software Center
✓Monitoring → Deployments shows Success count increasing
Troubleshooting
CONTENT_MISSING
Content not distributed to DP or DP in wrong boundary group. Check datatransferservice.log.
APP_REINSTALLS
Detection method wrong. Check AppDiscovery.log on client.
NOT_IN_SW_CENTER
Deployment type set to System but check User Experience settings. Check collection membership.
INSTALL_FAILS
Check AppEnforce.log on client for exact error and return code.
REQUIREMENTS
AppDiscovery.log shows requirements evaluation. Check OS version, disk space, arch rules.
Q&A
Q1 — Three types of detection methods?▾
File/Folder (check if specific file exists), Registry (check registry key/value), MSI Product Code (check Windows Installer DB for GUID).
Q2 — Difference between Required and Available deployment?▾
Required = forced install on schedule, no user choice. Available = appears in Software Center for user to install voluntarily.
Q3 — Log for application installation enforcement on client?▾
AppEnforce.log at C:\Windows\CCM\Logs\. Shows install command run, return code, detection result post-install.
Q4 — What must happen before deploying an application?▾
Content must be distributed to a DP first. Deploying without content = clients fail with "content unavailable."
Q5 — Where do you monitor per-device deployment status?▾
Monitoring workspace → Deployments → select deployment → Asset Details tab. Shows status per device: Success, In Progress, Error, Requirements Not Met.
Q6 — What is a Deployment Type?▾
The actual install logic within an Application. An Application can have multiple Deployment Types for different scenarios (x64 vs x86, MSI vs EXE). MECM evaluates requirements to pick the right one.
Q7 — What does Simulate Deployment do?▾
Runs detection method evaluation only — no actual install. Verifies detection logic before deploying to production.
Q8 — What is Supersedence?▾
Links new app version to old. When new version deploys, it auto-uninstalls the superseded version first. Configure during upgrades to prevent both versions co-existing.
Q9 — Log for why a client isn't detecting an installed app?▾
AppDiscovery.log — shows detection method evaluation results, exactly what MECM checked and found.
Q10 — Where is content status monitored after distributing to a DP?▾
Monitoring workspace → Distribution Status → Content Status. Shows per-DP state. Also check distmgr.log on site server.

Patch Management (SUP/WSUS)
7 / 10 difficulty
What it is

Patch Management uses the Software Update Point (SUP) — a WSUS wrapper — to sync, deploy, and enforce patches. MECM owns WSUS entirely; never approve updates in the WSUS console directly. Flow: sync updates from Microsoft → filter → create Software Update Groups → deploy via Deployment Packages. Automatic Deployment Rules (ADRs) automate recurring Patch Tuesday cycles.

Lifecycle placement
Pre-stage

Install WSUS role, configure SUP in MECM, set sync schedule and classifications

Stage

Run initial sync, review synced updates, create Software Update Groups

Deploy

Create Deployment Package, deploy to pilot then production. Set ADRs.

Verify

Monitor compliance, review non-compliant devices, check scan errors

Key concepts
SUP
Software Update Point
MECM role that wraps WSUS. Syncs from Microsoft, pushes policy to clients.
SUG
Software Update Group
Container for a set of updates to deploy together. Like a patch bundle.
Deployment Package
Content
Downloads update binaries and stores on UNC share for DP distribution.
ADR
Auto Deployment Rule
Automatically creates SUGs and deployments on schedule. Patch Tuesday automation.
Compliance
State reporting
Clients report installed updates. Compliance = % of required updates installed.
Deploy Patch Tuesday — steps
STEP 1
Software Library → Software Updates → Synchronize Software Updates after Patch Tuesday
STEP 2
Filter current month's Critical + Security updates for your OS versions
STEP 3
Create Software Update Group → add filtered updates
STEP 4
Download updates: right-click SUG → Download → create Deployment Package → UNC share on DP
STEP 5
Deploy to Pilot collection first — Required, deadline 7 days out
STEP 6
Monitor pilot compliance for 48-72 hours. Verify no breaking issues.
STEP 7
Deploy same SUG to Production — Required, deadline 14 days out, respect maintenance windows
STEP 8
Monitor Monitoring → Software Update Deployments for compliance %
Key checks
✓SUP sync completed — Software Library → Software Updates → Last Sync Time
✓Clients scanning for updates — check WUAHandler.log on client
✓Deployment Package distributed to DPs
✓Compliance rising over time in Monitoring → Deployments
Troubleshooting
SYNC_FAILS
Check wsyncmgr.log. Common: WSUS not connected to internet, SSL cert mismatch, WSUS DB corrupted.
NOT_SCANNING
Check WUAHandler.log and ScanAgent.log. SUP not in boundary group, or WSUS port mismatch (8530/8531).
0%_COMPLIANCE
Content not on DP, or clients haven't received policy. Check UpdatesDeployment.log on client.
WSUS_CONFLICT
Never approve/decline in WSUS console — MECM owns it. Manual changes cause sync conflicts.
Q&A
Q1 — What is an ADR and what problem does it solve?▾
Automatic Deployment Rule. Auto-creates SUGs, downloads updates, deploys on schedule. Eliminates manual Patch Tuesday process.
Q2 — Log to check if a client is scanning for updates?▾
WUAHandler.log — Windows Update Agent scan activity. Also ScanAgent.log for scan initiation logic.
Q3 — What is a Software Update Group?▾
A container grouping specific updates for deployment. Think patch bundle for a given month. You deploy the SUG to a collection, not individual updates.
Q4 — WSUS/SUP default ports?▾
HTTP: 8530, HTTPS: 8531. Must be open between clients and the SUP. #1 cause of clients not scanning.
Q5 — What happens if you approve updates in the WSUS console?▾
MECM owns WSUS — direct changes cause sync conflicts and break MECM's update management. Never approve/decline in WSUS when MECM manages it.
Q6 — What is a Deployment Package?▾
Container holding actual update binary files. Downloads from Microsoft and stores on UNC share for DP distribution. Multiple SUGs can share one Deployment Package.
Q7 — Log file for SUP sync failures?▾
wsyncmgr.log on the site server. Shows full sync cycle — categories, errors, duration.
Q8 — What update classifications to sync for standard Windows?▾
Critical Updates, Security Updates, Update Rollups, Service Packs, Definition Updates. Avoid Driver updates unless managing drivers via MECM.
Q9 — Where do you see per-device patch compliance?▾
Monitoring → Software Update Deployments → select deployment → right-click → View Status. Per-device: Compliant, Non-Compliant, Error, Unknown.
Q10 — Log showing update deployment enforcement on a client?▾
UpdatesDeployment.log on client. Shows updates in scope, enforcement state, reboot behavior. Also WindowsUpdate.log for WUA install activity.

Operating System Deployment (OSD)
9 / 10 difficulty
What it is

OSD automates full OS installations via Task Sequences. A Task Sequence is a series of ordered steps executed in WinPE or Windows: partition disk, apply image, install drivers, install software, join domain. OSD is the hardest module because it chains many dependencies: boot images, OS images, driver packages, task sequences, PXE, and boundary groups must all be correct. One broken link silently fails the whole deployment.

Lifecycle placement
Pre-stage

Configure PXE on DP, build boot image, capture/import OS image, create driver packages

Stage

Build task sequence, test on single device, validate all steps

Deploy

Deploy TS to All Unknown Computers for bare metal; device collection for refresh

Verify

Check SMSTS.log, domain join, client registered, apps installed

Key components
Boot Image
WinPE
Minimal Windows PE loaded before OS install. Must match ADK version.
OS Image
WIM file
Captured or Microsoft-provided Windows image. Applied to disk during TS.
Task Sequence
TS
Ordered steps MECM executes. Can be fully automated or prompt for input.
Driver Package
Hardware
Drivers injected during WinPE or post-OS phase. Must match hardware model.
PXE
Network boot
Boots devices over network to receive TS. Enable on DP. Requires DHCP options 66/67.
Unknown Computers
Bare metal
Built-in collection for devices not yet in MECM DB.
Bare metal TS — steps
STEP 0
Patch site server and all VMs. Confirm ADK version matches MECM version before building boot image.
STEP 1
Enable PXE on DP — Administration → Distribution Points → DP Properties → PXE tab
STEP 2
Distribute Boot Image to PXE-enabled DP
STEP 3
Import OS Image (.wim) — Software Library → OS → Operating System Images → Add
STEP 4
Create Driver Package for target hardware — import INF drivers, distribute to DP
STEP 5
Create Task Sequence: Software Library → OS → Task Sequences → Create → Install existing image package
STEP 6
Edit TS: Partition Disk → Apply OS → Apply Windows Settings → Apply Drivers → Setup Windows and ConfigMgr → Install Software → Domain Join
STEP 7
Deploy TS to All Unknown Computers — Available for testing, Required for production
STEP 8
PXE boot target device — F12 on physical, change boot order in VM BIOS
STEP 9
Monitor: SMSTS.log in WinPE at X:\Windows\Temp\SMSTSLog\
Key checks
✓Boot image deployed to PXE-enabled DP before any PXE boot attempt
✓DHCP scope options 66 (TFTP server) and 67 (boot file) configured
✓Device shows in MECM console after TS with correct site assignment
✓Domain join step succeeded — device in correct OU in AD
✓SMSTS.log shows no errors, TS exit code 0
Troubleshooting
PXE_NO_BOOT
DHCP options 66/67 missing, boot image not on DP, or PXE not enabled. Check pxecontrol.log.
TS_FAILS_WINPE
Read SMSTS.log at X:\Windows\Temp\SMSTSLog\. Press F8 for command prompt.
DRIVER_MISSING
NIC/storage driver missing in WinPE. Import correct INF driver, add Apply Driver Package step with hardware model condition.
DOMAIN_JOIN_FAILS
Service account needs rights to create computer objects in target OU. Check DNS in WinPE first.
CLIENT_NOT_REG
Setup Windows and ConfigMgr step — verify correct site code and MP. Check boundary group covers deployment subnet.
Q&A
Q1 — What is SMSTS.log and where is it during WinPE?▾
Master Task Sequence log — records every step, success/failure, error codes. In WinPE: X:\Windows\Temp\SMSTSLog\SMSTS.log. After OS applies: C:\Windows\CCM\Logs\SMSTS.log.
Q2 — What are DHCP scope options 66 and 67?▾
Option 66 = TFTP boot server IP (your PXE-enabled DP). Option 67 = boot file name. Required for PXE — without these DHCP hands out an IP but doesn't tell the client where to boot from.
Q3 — How do you get a command prompt during WinPE?▾
Press F8 (must be enabled in boot image properties → enable command support). Opens cmd for navigating to SMSTS.log and running diagnostics.
Q4 — Collection for bare metal deployments?▾
All Unknown Computers — built-in collection for devices that PXE booted but aren't yet in the MECM DB.
Q5 — Difference between OS Image and Boot Image?▾
Boot Image = WinPE — loads first via PXE, runs TS engine, partitions disk. OS Image = full Windows WIM applied to disk. Both required.
Q6 — What TS step installs the MECM client?▾
Setup Windows and ConfigMgr — installs CCMSetup, specifies site code and MP, initiates client registration. Built-in TS step.
Q7 — ADK components required for OSD?▾
Windows ADK (Deployment Tools component) and WinPE Add-on. ADK version must match MECM version's supported matrix.
Q8 — TS variable that controls restart at end?▾
SMSTSRebootRequested set to TRUE triggers restart. OSDRebootWaitTime controls the countdown timer.
Q9 — How do you inject drivers for a specific hardware model?▾
Create a Driver Package for that model, add Apply Driver Package step in TS with WMI condition: SELECT * FROM Win32_ComputerSystem WHERE Model LIKE '%YourModel%'
Q10 — What is prestaged media and when to use it?▾
A WIM written to a hard drive/USB before deployment. Used when PXE isn't available (remote sites, no network boot). Device boots from media, then contacts MECM for policy.
MODULE 07 / 10StageDeployVerify
Compliance Settings (DCM)
6 / 10 difficulty
What it is

Desired Configuration Management (DCM) lets you define settings devices should have — registry keys, file content, script outputs, WMI values — and measure compliance. Configuration Items (CIs) define the check. Configuration Baselines group CIs and are deployed to collections. Clients evaluate and report Compliant or Non-Compliant. DCM can also remediate automatically by pushing the correct value when a check fails.

Lifecycle placement
Stage

Define Configuration Items, group into Baselines

Deploy

Deploy baselines to device collections, enable remediation

Verify

Review compliance reports, identify non-compliant devices, investigate CI failures

Key concepts
Configuration Item
CI
A single compliance check — registry value, script result, file exists, WMI query.
Configuration Baseline
CB
Collection of CIs deployed together. The unit you deploy to a collection.
Remediation
Auto-fix
When non-compliant, MECM applies the correct value. Enable per CI and per deployment.
Setting Types
Flexibility
Registry, File, Script (PowerShell), WMI, IIS, SQL, XML, LDAP.
Create CI + Baseline
STEP 1
Assets and Compliance → Compliance Settings → Configuration Items → Create Configuration Item
STEP 2
Select type: Windows Desktops and Servers. Name clearly: e.g., "Check - RDP Enabled"
STEP 3
Add Setting: Registry → HKLM\System\CurrentControlSet\Control\Terminal Server, value fDenyTSConnections
STEP 4
Set Compliance Rule: value must equal 0. Enable Remediate if non-compliant.
STEP 5
Create Configuration Baseline → Add CI → save
STEP 6
Deploy Baseline to collection → enable Remediate non-compliant → set evaluation schedule
STEP 7
Monitor: Assets and Compliance → Deployments → select baseline → View Status
Key checks
✓Baseline deployed and evaluation cycle has run at least once
✓Compliant % tracking upward after remediation enabled
✓Non-compliant devices — open device properties → Configurations tab for per-CI results
Troubleshooting
ALL_NONCOMPLIANT
Setting path or value wrong. Test registry path manually on sample device. Check DCMAgent.log.
REMED_NOT_WORK
Remediation must be enabled both on CI setting AND on baseline deployment. Check both locations.
SCRIPT_CI_FAILS
PowerShell must return specific value. Check execution policy and output format. Check DCMAgent.log.
Q&A
Q1 — Difference between CI and Baseline?▾
CI = single compliance check (one registry key). Baseline = collection of CIs grouped together. You deploy Baselines to collections, not individual CIs.
Q2 — Where do you see per-device CI compliance results?▾
Assets and Compliance → Devices → open device properties → Configurations tab. Shows each deployed baseline and per-CI state.
Q3 — Log file on client for DCM evaluation?▾
DCMAgent.log — shows baseline evaluation, CI checks, remediation attempts, results.
Q4 — What setting types can a CI check?▾
Registry, File/Folder, Script (PowerShell, VBScript, JScript), WMI, IIS Metabase, SQL Server query, XML file, LDAP, Assembly.
Q5 — How to enable auto-remediation?▾
Two places: (1) CI Compliance Rule — check "Remediate noncompliant rules when supported." (2) Baseline Deployment — enable same setting. Both must be checked.
Q6 — Can a script CI write back to the device?▾
Yes — separate Discovery script (read check) and Remediation script (applies fix). If non-compliant, MECM runs the remediation script.
Q7 — Default evaluation schedule?▾
7 days. Can go as low as 1 hour for critical checks. Force immediate: Control Panel → Configuration Manager → Configurations tab → Evaluate.
Q8 — Where are Configuration Baselines in the console?▾
Assets and Compliance workspace → Compliance Settings → Configuration Baselines.
Q9 — Can you import pre-built CIs from Microsoft?▾
Yes — Microsoft publishes Security Compliance Baselines as .cab files importable via Import Configuration Data. Saves building from scratch.
Q10 — Difference between Compliant, Non-Compliant, and Error?▾
Compliant = all rules met. Non-Compliant = at least one rule failed. Error = CI evaluation itself failed (script error, WMI timeout, permission denied) — fix errors before trusting compliance numbers.

Boundaries & Boundary Groups
6 / 10 difficulty
What it is

Boundaries and Boundary Groups define which site systems (MPs, DPs, SUPs) a client uses based on its network location. A Boundary is a network definition. A Boundary Group links boundaries to site systems. Clients match their IP to a Boundary, find the Boundary Group, and connect to the Site Systems in that group. Misconfigured boundary groups are the #1 silent failure in MECM — clients register but pull from the wrong DP or fall back to slow links.

Lifecycle placement
Pre-stage

Map network subnets to boundaries before installing clients

Stage

Create boundary groups, assign site systems, configure fallback

Verify

Confirm clients use correct MP and DP, no unexpected fallback

Key concepts
Boundary
Network definition
IP Subnet, IP Range, AD Site, or IPv6 Prefix.
Boundary Group
Logical container
Links boundaries to site systems. Clients use systems in their matched group.
Default BG
Fallback
Built-in catch-all. Don't add DPs here without intent.
Fallback
Neighbor BG
Configure neighbor boundary groups for content fallback. Set fallback time.
Site Assignment
Primary Site
Boundary group can determine which site a client is assigned to.
Configure boundary groups
STEP 1
Administration → Hierarchy Configuration → Boundaries → Create Boundary → IP Subnet
STEP 2
Administration → Boundary Groups → Create Boundary Group
STEP 3
Boundaries tab: add the subnet boundaries that belong to this group
STEP 4
References tab: add Site Systems — select your MP, DP, and SUP
STEP 5
Check "Use this boundary group for site assignment" if clients should auto-assign here
STEP 6
Configure fallback: Relationships tab → add neighbor boundary group with fallback time (e.g., 60 min)
STEP 7
Test: install client on subnet device, verify correct MP and DP via locationservices.log
Key checks
✓Every client subnet covered by at least one boundary — no orphaned subnets
✓Each boundary group has at least one MP and one DP assigned
✓LocationServices.log shows correct MP selected for subnet
✓Default Boundary Group does not contain DPs unless intentional
Troubleshooting
NO_MATCH
Client IP not in any boundary — falls to Default BG. Check locationservices.log for "No matching boundary."
WRONG_DP
Multiple overlapping boundary groups for same subnet. Check for subnet conflicts in Administration → Boundaries.
CONTENT_NOT_FOUND
Content on DP but not in client's boundary group. Verify DP is in correct boundary group References tab.
SLOW_FALLBACK
Fallback time too long — reduce for fast-link environments.
Q&A
Q1 — Difference between a Boundary and a Boundary Group?▾
Boundary = network definition. Boundary Group = container linking boundaries to site systems. A Boundary alone does nothing — must be in a group.
Q2 — Four boundary types?▾
IP Subnet, IP Address Range (start-end), Active Directory Site name, IPv6 Prefix.
Q3 — What happens if a client's IP doesn't match any boundary?▾
Falls to Default Site Boundary Group. If that group has no site systems, client may register but can't find a local DP — content fails.
Q4 — Log to check which MP a client selected?▾
LocationServices.log on the client. Shows boundary evaluation, MP selection, and DP location requests.
Q5 — Where do you assign a DP to a boundary group?▾
Administration → Boundary Groups → open group → References tab → Add → select the DP.
Q6 — What is the Default Site Boundary Group?▾
Built-in catch-all. Clients not matching any other group fall here. Also the fallback target for neighbor relationships. Don't add DPs here unless you want all unmatched clients downloading from them.
Q7 — What is a fallback relationship?▾
A configured neighbor group that a client tries for content after a time threshold. Configured in Boundary Group → Relationships tab.
Q8 — Can one boundary belong to multiple boundary groups?▾
Yes — but causes overlap. Client matches multiple groups and MECM uses all matched site systems, causing unexpected DP selection. Keep boundaries in one group unless intentional.
Q9 — Setting in a boundary group for automatic site assignment?▾
General tab → check "Use this boundary group for site assignment."
Q10 — For your lab (mypdomain 172.16.0.x, mybdomain 191.16.0.x) — minimum boundary setup?▾
One Boundary per subnet. One Boundary Group per domain linking each subnet to that domain's MP, DP, and SUP. Separate groups prevent cross-domain content routing.

SQL & the CM Database
8 / 10 difficulty
What it is

The MECM site database stores everything — device inventory, deployments, compliance, collections, patch status, and site config. SQL health equals MECM health. A slow or full SQL DB causes console slowness, deployment failures, and inaccurate reports. You don't need to be a DBA, but you need the key views, maintenance tasks, sizing rules, backup strategy, and how to query for operational data.

Lifecycle placement
Pre-stage

Size SQL VM correctly, configure collation, set memory limits, plan backup

Stage

Install SQL, configure maintenance, enable CM DB backup task

Verify

Monitor DB size, fragmentation, backup jobs, SQL error log

SQL requirements & sizing
Collation
Required
SQL_Latin1_General_CP1_CI_AS — set at SQL install. Non-negotiable.
SQL Memory
Cap it
Set max memory = total RAM minus 4-8GB. Default unlimited starves OS.
DB Files
Placement
Separate .mdf and .ldf onto different disks where possible.
CM DB Name
Convention
Default: CM_[SiteCode] e.g., CM_P01. Never rename after creation.
SQL Version
MECM 2509
SQL Server 2022 required.
TempDB
Performance
Set TempDB file count = CPU core count (max 8). Reduces allocation contention.
Key operational views
VIEW
v_R_System — all discovered devices. Core join table via ResourceID.
VIEW
v_GS_INSTALLED_SOFTWARE — all installed software per device.
VIEW
v_GS_PROCESSOR, v_GS_PHYSICAL_MEMORY, v_GS_DISK — hardware inventory per device.
VIEW
v_UpdateComplianceStatus — patch compliance per device per update.
VIEW
v_DeploymentSummary — deployment results summary by deployment ID.
VIEW
v_ClientDeploymentState — client health and deployment state per device.
Maintenance tasks
STEP 1
Enable built-in maintenance tasks: Administration → Site Configuration → Sites → right-click → Site Maintenance
STEP 2
Enable Backup Site Server and all Delete Aged Data tasks
STEP 3
Configure SQL maintenance: weekly index rebuild, nightly reorganize, daily stats update (Ola Hallengren scripts recommended)
STEP 4
Monitor DB size in SSMS. Set transaction log to Simple recovery if full backups are in place.
STEP 5
Review SQL Error Log weekly — look for severity 17+ errors, deadlocks, blocking.
Key checks
✓SQL collation confirmed before MECM install — verify in SSMS: Server Properties → General
✓SQL max memory set — Server Properties → Memory
✓Backup Site Server maintenance task enabled and running successfully
✓Delete Aged Data tasks all enabled to prevent DB bloat
Troubleshooting
CONSOLE_SLOW
Index fragmentation. Run index maintenance. Check SQL blocking via sys.dm_exec_requests.
DB_GROWING
Aged data tasks not running. Enable all Delete Aged Data tasks. Check transaction log growth.
BACKUP_FAILS
Monitoring → Site Status → Backup Site Server component. Check SQL Agent job history.
SQL_MEMORY
OS starved — SQL took all RAM. SSMS → Server Properties → Memory → set Max Server Memory.
Q&A
Q1 — SQL collation required and where set?▾
SQL_Latin1_General_CP1_CI_AS. Set during SQL Server installation at instance level. Cannot be changed without rebuilding SQL.
Q2 — Default CM database name?▾
CM_[SiteCode] e.g., CM_P01. Created automatically during MECM setup. Never rename after creation.
Q3 — Why must SQL max memory be capped?▾
SQL defaults to unlimited and consumes all RAM, starving the OS. Set max server memory = total RAM minus 4-8GB. Via SSMS: Server Properties → Memory.
Q4 — Primary join key for device data across views?▾
v_R_System.ResourceID — used in almost all queries: JOIN v_R_System sys ON other_view.ResourceID = sys.ResourceID
Q5 — Where in console do you enable built-in maintenance tasks?▾
Administration → Site Configuration → Sites → right-click site → Site Maintenance.
Q6 — What causes console to become slow over time?▾
Index fragmentation from heavy write activity. Run weekly index rebuilds. Also: aged data not purged and SQL memory not limited.
Q7 — What does v_GS_INSTALLED_SOFTWARE contain?▾
Software inventory — all programs detected via Add/Remove Programs and WMI. Columns: ProductName, ProductVersion, Publisher, InstallDate. Join to v_R_System via ResourceID.
Q8 — Raw tables or v_ views for queries?▾
Always v_ views — Microsoft-supported, documented, stable across MECM versions. Raw tables change schema between releases and are unsupported.
Q9 — Recommended TempDB configuration?▾
One TempDB data file per CPU core, max 8 files, all equal size. Reduces allocation page contention under heavy MECM write loads.
Q10 — How to back up the MECM site?▾
Enable Backup Site Server maintenance task in MECM — backs up site DB, site files, and CD.Latest folder. Also use SQL native backups for the CM DB. Test restores periodically.
